Write the initial product brief for a new initiative. Run `/office-hours` on the raw idea first — write the brief only after office-hours confirms the direction.

## Required fields

**Standard fields:**

- **Target user** — who specifically? Not "users" — name the persona, context, and job to be done
- **Problem** — what pain or desire are we solving? What is the user doing today instead?
- **Scope** — what is in for this release?
- **Non-goals** — what are we explicitly not building?
- **Success criteria** — what must be true for this to be considered shipped successfully?
- **Platform wedge** — web, mobile, macOS, or a narrower first surface?

**B2C required fields:**

- **Activation metric** — what is the one concrete user action that proves they received value? Not a pageview or a session. A specific, measurable action (e.g., "user creates their first project", "user sends their first message", "user completes onboarding checklist"). The Weekly Portfolio Review will track this metric for every shipped feature.
- **Distribution vector** — how will new users discover this feature or product? Word of mouth, SEO, in-app prompt, push notification, email, paid? If the answer is "they'll just find it," that is not a distribution plan — revisit before proceeding.
- **Retention mechanic** — why does the user come back after day 1? What is the trigger (habit, notification, value loop)? A feature without a retention mechanic produces one-time activations, not engaged users.
- **Delight moment** — what is the specific moment in the experience that would make a user tell a friend? If you cannot name it, the feature probably does not have one — find it or build it in.

## Done means

The Senior Designer can start the experience spec without guessing who the user is, what success looks like, or what the B2C growth and retention model is.
